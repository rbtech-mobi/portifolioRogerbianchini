SaaS backend and PWA

Multi Tenant Ordering Platform

A reusable ordering and operations platform for independent food businesses. Each tenant has its own catalog, orders, users, branding and payment records. Access is scoped on the server for every protected request.

11 database tables 13 automated tests Node.js and MySQL Firebase authentication
Technical notes

Problem

A shared platform reduces deployment and maintenance work, but it creates a strict requirement: one business must never read or change another business's data.

What I built

A public catalog, cart and order flow for customers, plus an installable admin PWA for menus, order states, payments and branding.

Request flow

ExpressRequest ID, security headers, rate limits and body limits.
IdentityFirebase verifies the ID token and adds the authenticated user to the request.
Tenant contextAn admin user lookup resolves the business ID before protected data is queried.
Service layerOrder rules, state transitions, transactions and payment ledger logic.
RepositoryParameterized SQL with business ID scoping on tenant owned records.

Key decisions

  • Money is stored as integer cents.
  • Payments use an append only ledger.
  • Order creation supports idempotency.
  • Uploaded images are decoded and rewritten before storage.

Stack

Node.js, Express 5, MySQL, Firebase Auth, Zod, Sharp, Helmet, PWA, Service Worker, vanilla JavaScript, node:test and Supertest.

Sales workflow and LLM integration

Outreach Calibrator

A prospecting assistant that combines CRM fields and research notes to produce a more relevant first message. It is designed to fit inside an SDR workflow rather than operate as a separate writing tool.

monday.com webhooks LLM integration Structured prompts Workflow automation
Technical notes

Problem

Personalized outreach requires research. High volume outreach often removes that context and produces messages that sound generic.

What I built

A small application that receives prospect data from monday.com or manual entry, combines it with free text research, and sends a structured request to an LLM.

Workflow

InputProspect, role, company, website, offer and research notes.
Ingestionmonday.com webhook or native automation, with manual entry as a fallback.
AnalysisThe prompt maps the prospect context to the seller's value proposition.
OutputA suggested approach angle and a first message that the representative can review and edit.

Why it is useful

The application reduces the time needed to turn account research into an outreach draft while keeping the representative responsible for the final message.

Stack

LLM integration, monday.com webhooks, structured JSON payloads, workflow automation and prompt design.

Embedded device and MQTT workflow

Desktop Workflow Terminal

A touchscreen device for time sensitive approvals. It receives a workflow message over encrypted MQTT, displays the request, and publishes the selected response back to the automation server.

ESP32 S3 C++ and FreeRTOS MQTT over TLS n8n orchestration
Technical notes

Problem

Some approvals are easy to miss in busy chat channels and inboxes. A dedicated device makes the request visible and gives the operator a direct response path.

What I built

Firmware for a WT32 SC01 Plus with a 3.5 inch touch display, dual WiFi failover, MQTT messaging, device health status and a simple response interface.

Device flow

ConnectivityPrimary and secondary WiFi networks with automatic recovery.
MessagingHiveMQ Cloud over TLS, with QoS 1 for incoming and outgoing messages.
InterfaceTouch input through I2C and display rendering with LovyanGFX.
Workflown8n publishes requests and consumes the selected response.
HealthA retained status topic reports device availability on a fixed interval.

Technical scope

The project covers embedded state management, encrypted messaging, reconnection logic, touch input and a physical interface for a software workflow.

Stack

ESP32 S3, C++, FreeRTOS, MQTT, TLS, LovyanGFX, ArduinoJson, HiveMQ Cloud and n8n.
