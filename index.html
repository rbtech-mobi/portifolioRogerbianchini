Objetivo da disciplina: Avaliar a capacidade dos acadêmicos em criar e manipular Interfaces Gráficas utilizando o Jetpack Compose.

Além do Enunciado

Do trabalho acadêmico ao padrão de mercado

Encarei o enunciado não como um checklist a cumprir, mas como ponto de partida para entregar um produto que eu me orgulharia de colocar em produção. As decisões abaixo não foram exigidas — foram deliberadas.

🌐
Internacionalização Dinâmica (i18n)
PT-BR · EN-US · Runtime Locale Switching
O enunciado não pedia múltiplos idiomas. Implementei suporte completo PT-BR / EN-US com troca em tempo de execução via AppCompatDelegate.setApplicationLocales(), sem reiniciar o app. Nenhum texto está hardcoded — 100% via stringResource().
📐
Clean Architecture · UDF
MVVM · StateFlow · State Down / Events Up
Refatorei completamente a estrutura para seguir UDF (Unidirectional Data Flow). Nenhum @Composable gerencia lógica de negócio. Todo estado flui do TransactionViewModel via StateFlow; a UI só emite eventos de volta.
🇧🇷
Regra de Ouro da Moeda
Sempre R$ · Locale pt-BR fixo
Independente do idioma selecionado, a formatação financeira é sempre Real Brasileiro: separador de milhar com ponto, decimal com vírgula, símbolo R$. Implementado via NumberFormat.getCurrencyInstance(Locale("pt", "BR")) fixo.
🔤
Nomenclatura 100% em Inglês
Clean Code · Padrão de Mercado
O enunciado entregava código misto: Lancamento, dataPrevisao, salvar(). Renomeei tudo: Transaction, dueDate, save(). Português vive exclusivamente nos arquivos strings.xml.
🎨
Design System Dark Premium
Gradiente · Tipografia ampliada · Alto contraste
Fundo com Brush.verticalGradient (DarkBlue → Purple). Tipografia 16sp+ em textos base, 24sp+ em títulos. Nenhum ícone ou texto principal em preto — uso de Color.White e cores de alto contraste em todo o sistema.
📅
Modelo de Previsão vs. Realização
dueDate · paymentDate · isCompleted
Além do simples checkbox "pago", implementei o modelo duplo: dueDate (vencimento previsto) e paymentDate (data de realização). O botão animado "Marcar como Concluído" abre um DatePickerDialog para registrar a data exata do movimento financeiro.
Decisão Técnica

Por que utilizei XML em um projeto Compose?

Jetpack Compose substitui os arquivos XML de layout (activity_main.xml, fragment_list.xml). Mas a plataforma Android ainda usa XML em camadas que o Compose não alcança — e entender essa distinção é o que separa um desenvolvedor Android de um desenvolvedor que apenas usa o Compose.

Regra de ouro: Compose substitui XML de layout de UI. Não substitui o sistema de recursos da plataforma Android, que continua sendo declarado em XML e consumido pelo runtime do Android antes mesmo do Compose ser inicializado.
✗ XML que SUBSTITUÍ com Compose
Layouts de interface — eliminados completamente
✗
res/layout/activity_main.xml
Definição de UI declarativa — substituído por @Composable functions. O app não tem nenhum arquivo de layout XML.
✗
res/layout/fragment_list.xml
Fragments com XML — substituídos por TransactionListScreen, TransactionFormScreen como Composables stateless.
✗
RecyclerView + Adapter XML
Substituído por LazyColumn { items(...) { TransactionListItem() } } — sem adapter, sem ViewHolder, sem DiffUtil manual.
✓ XML que MANTIVE (plataforma Android)
Recursos de sistema — Compose os consome, não os substitui
✓
res/values/strings.xml
Sistema de recursos de strings do Android. Obrigatório para i18n com qualificadores de locale (values-en/). Compose acessa via stringResource(R.string.x).
stringResource(R.string.save)
✓
res/xml/locales_config.xml
Exigido desde Android 13 (API 33) para o atributo android:localeConfig no Manifest. Sem ele, AppCompatDelegate.setApplicationLocales() não funciona corretamente. Não existe API Kotlin/Compose equivalente.
android:localeConfig="@xml/locales_config"
✓
AndroidManifest.xml
Contrato da aplicação com o sistema operacional Android. Define Activity, permissões, ícone, tema, suporte a RTL e configuração de locale. É um requisito da plataforma — não é UI.
✓
drawable/ic_launcher_*.xml
Adaptive Icons do Android usam vector drawables XML em camadas (background + foreground). O sistema mipmap renderiza o ícone do launcher antes do runtime do Compose inicializar — não há API Compose para isso.
<adaptive-icon> → mipmap-anydpi/

locales_config.xml — Sistema de Idiomas

res/xml/locales_config.xml 
<locale-config xmlns:android="...">
   <locale android:name="pt-BR"/>
   <locale android:name="en-US"/>
</locale-config>

// No Manifest:
<application
    android:localeConfig="@xml/locales_config"
    .../>

Adaptive Icon XML: Os arquivos ic_launcher_background.xml e ic_launcher_foreground.xml são vector drawables que representam as camadas do ícone adaptativo (API 26+). Arquitetura

UDF — Unidirectional Data Flow

A arquitetura segue o padrão recomendado pelo Android para aplicações Compose: estado desce do ViewModel para a UI, eventos sobem da UI de volta para o ViewModel. Nenhum Composable contém lógica de negócio ou acessa dados diretamente.

Kotlin · TransactionViewModel.kt — dashboardState Flow 
// Estado calculado reativamente a partir da lista de transações
val dashboardState = _transactions.map { list ->
    val completedIncome = list
        .filter { it.type == TransactionType.INCOME && it.isCompleted }
        .sumOf { it.amount }
    val completedExpenses = list
        .filter { it.type == TransactionType.EXPENSE && it.isCompleted }
        .sumOf { it.amount }
    val toReceive = list.filter { it.type == INCOME && !it.isCompleted }.sumOf { it.amount }
    val toPay    = list.filter { it.type == EXPENSE && !it.isCompleted }.sumOf { it.amount }
    val currentBalance   = completedIncome - completedExpenses
    val projectedBalance = currentBalance + toReceive - toPay

    DashboardState(currentBalance, toReceive, toPay, projectedBalance)
}

// Na UI — State Down: o Composable só observa, nunca calcula
val state by viewModel.dashboardState.collectAsState(initial = null)
Implementação

Principais Decisões Técnicas

TransactionListItem — Layout conforme enunciado

O item da lista implementa exatamente o layout especificado: ícone à esquerda com cor por tipo, linha 1 apenas com descrição, linha 2 com data à esquerda e valor com sinal negativo à direita.

Kotlin · TransactionListItem.kt 
val typeColor  = if (transaction.type == TransactionType.INCOME) ColorIncome else ColorExpense
val amountText = (if (transaction.type == TransactionType.EXPENSE) "-" else "") +
                 transaction.amount.formatAsBrazilianCurrency()

Row(...) {
    // Ícone à esquerda — cor por TIPO, forma por STATUS
    Icon(
        imageVector = if (transaction.isCompleted) Icons.Default.CheckCircle
                      else Icons.Default.Schedule,
        tint = typeColor,   // Color(0xFF00984E) ou Color(0xFFCF5355)
        modifier = Modifier.size(32.dp)
    )
    Column(modifier = Modifier.weight(1f)) {
        // Linha 1 — apenas descrição
        Text(text = transaction.description, color = Color.White, maxLines = 1)
        // Linha 2 — data à esq, valor à dir (com sinal negativo para despesas)
        Row(horizontalArrangement = Arrangement.SpaceBetween) {
            Text(text = transaction.dueDate.format(dateFormatter), color = Color.White.copy(alpha = .7f))
            Text(text = amountText, color = typeColor, fontWeight = FontWeight.ExtraBold)
        }
    }
}

Validação do Campo Valor — Proteção contra Crash

Kotlin · TransactionFormScreen.kt — input guard 
// Filtro na entrada: aceita apenas dígitos e um único ponto decimal
// Previne crash ao receber letras, vírgulas, múltiplos pontos, etc.
onValueChange = { input ->
    if (input.isEmpty() || input.matches(Regex("^\\d*\\.?\\d*$"))) {
        amountStr = input
    }
    amountError = false
}

// Validação no save: campo obrigatório + parseável como Double
val amount = amountStr.toDoubleOrNull()
descriptionError = description.isBlank()
amountError      = amountStr.isBlank() || amount == null

if (!descriptionError && !amountError) {
    onSave(Transaction(amount = amount ?: 0.0, ...))
}

Design System

Sistema Visual — Cores e Decisões

O design ignora o tema claro padrão do Material 3. Toda a interface é construída sobre um fundo de gradiente escuro (DarkBlue → Purple), com tipografia ampliada e cores semânticas que comunicam status financeiro de forma imediata e inequívoca.

Racionalidade das cores: Verde (#00984E) e vermelho (#CF5355) são as cores especificadas no próprio enunciado da UTFPR — escolhidas por serem universalmente associadas a lucro e perda em sistemas financeiros. O gradiente DarkBlue → Purple comunica premium e confiança sem recorrer ao visual genérico de fintech blue. O âmbar para projeções sinaliza atenção / futuro sem alarme — convenção de dashboards financeiros.
Acessibilidade: Todos os textos usam Color.White sobre fundo escuro, garantindo contraste mínimo 4.5:1 (WCAG AA). Nenhum ícone ou texto principal usa preto — regra aplicada em todo o design system.

Adaptive Icon — Identidade do App

O ícone foi desenhado como vector drawable XML em duas camadas (background + foreground), seguindo o sistema adaptive icon do Android (API 26+). O background replica o gradiente do app (DarkBlue → Purple via <gradient> no aapt). O foreground traz as duas setas — verde subindo (Receita) e vermelha descendo (Despesa) — com um círculo branco representando a moeda. O Android OS aplica a máscara de forma do launcher automaticamente.

        android:endY="108">

Tech Stack

Frameworks e Justificativas

Cada biblioteca foi escolhida por maturidade, suporte oficial do Google e aderência ao padrão recomendado para Android moderno — não por familiaridade ou conveniência.

🚀
Jetpack Compose (BOM 2024.09)
Framework declarativo de UI do Google. Substitui XML de layouts, elimina findViewById, simplifica o estado com recomposição automática. Material 3 integrado nativamente.
ui · declarativo
⚙️
Kotlin 2.2.10 + K2 Compiler
Tipo-safe, null-safe, com suporte nativo a coroutines e Flow. O K2 (novo frontend do compilador Kotlin) melhora tempo de build e inferência de tipos — critical para evitar o erro de tipo nas lambdas do SettingsDialog.
language
🏛️
ViewModel + StateFlow
ViewModel sobrevive a rotação de tela e configuration changes. StateFlow é o canal reativo de estado — a UI observa via collectAsState(), nunca manipula estado diretamente.
arquitetura
🌐
AndroidX AppCompat 1.7.1
Necessário especificamente para o AppCompatDelegate.setApplicationLocales() — a única API que permite troca de idioma em runtime sem reiniciar o app. A Activity herda de AppCompatActivity por este motivo.
i18n
📅
java.time (API 26+)
LocalDate para representação de datas sem timezone, imutável e tipada. Elimina os bugs históricos de java.util.Date. Min SDK 26 foi definido exatamente para habilitar java.time sem desugaring.
utilities
🎨
Material Icons Extended
Biblioteca completa de ícones do Material Design (2000+). Usada para CheckCircle, Schedule, Settings, Add, ArrowBack, EditCalendar, Notes — todos os ícones do app sem assets externos.
ui
🏗️
AGP 9.1.0 + Gradle Wrapper
Android Gradle Plugin versão-lockada para builds reprodutíveis. Suporte a Compose Compiler via plugin aliasing (Kotlin Compose plugin). compileSdk 36 · targetSdk 36.
build
🔤
Sistema de Recursos Android (XML)
strings.xml (PT-BR + EN-US), locales_config.xml, AndroidManifest.xml, adaptive icon drawables. Camada de plataforma que o Compose consome mas não substitui — veja seção "XML no Compose".
plataforma
✅
JUnit 4 + AndroidX Test
Infraestrutura de testes unitários e instrumentados incluída na configuração base. A arquitetura UDF facilita testes unitários do ViewModel isolado da UI.
