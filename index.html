System Architecture

Layered MVVM Architecture

The system is structured across three independent tiers — the Android client, a locally-hosted REST API, and Firebase-managed cloud services — following a Model-View-ViewModel pattern that decouples the presentation layer from data concerns.

Identity & Access

Phone-Based Authentication

The system implements a stateless, token-based authentication model via Firebase Authentication — specifically using the Phone/OTP (One-Time Password) provider. The flow eliminates password management overhead and leverages the device's SIM identity as a second factor by design.

Security posture: Firebase Authentication tokens are JSON Web Tokens (JWTs) signed with RS256. The client never stores raw credentials — only the short-lived ID token (1h TTL) and a long-lived refresh token managed by the Firebase SDK within the Android Keystore-backed secure storage.

Session & Token Lifecycle
JWT · RS256 · Firebase ID Token
→
ID Token (short-lived)
JWT signed with RS256. TTL of 1 hour. Contains uid, phone, iss, aud claims. Must be refreshed before expiry.
TTL: 3600s
→
Refresh Token (long-lived)
Opaque token stored securely by Firebase SDK. Used to issue new ID tokens without re-authentication. Revocable server-side.
→
Automatic token refresh
Firebase SDK monitors token expiry and transparently refreshes via getIdToken(forceRefresh=false). App state observer fires on AuthStateChanged.
FirebaseAuth.AuthStateListener
→
Logout (session invalidation)
FirebaseAuth.signOut() clears the local token cache and invalidates the session. AuthStateListener broadcasts the unauthenticated state; Activity stack is cleared.
auth.signOut() → startActivity(Login)
→
Persistent auth state
If the user closes and reopens the app, currentUser is non-null and the session resumes without re-authentication — until explicitly signed out or token revoked.
Kotlin · LoginActivity.kt — OTP dispatch 
// PhoneAuthOptions built with a 60s timeout; callbacks registered on calling Activity
val options = PhoneAuthOptions.newBuilder(auth)
    .setPhoneNumber(phoneNumber)
    .setTimeout(60L, TimeUnit.SECONDS)
    .setActivity(this)
    .setCallbacks(object : PhoneAuthProvider.OnVerificationStateChangedCallbacks() {

        override fun onVerificationCompleted(credential: PhoneAuthCredential) {
            // Auto-retrieval on supported devices (SMS Retriever API)
            signInWithCredential(credential)
        }

        override fun onCodeSent(
            verificationId: String,
            token: PhoneAuthProvider.ForceResendingToken
        ) {
            viewModel.setVerificationId(verificationId)  // persisted in ViewModel
            navigateToOtpScreen()
        }

        override fun onVerificationFailed(e: FirebaseException) {
            viewModel.setError(e.message)               // surfaced via LiveData
        }
    }).build()

API Reference

REST API — Vehicle Resource

The backend exposes a RESTful API following HTTP semantics strictly. All request/response bodies are application/json. Interactive documentation available at /api-docs (Swagger/OpenAPI 3.0).

Method Endpoint Description Request Body Success
GET /car Returns the full collection of registered vehicles. No pagination in v1. — 200 · Car[]
POST /car Persists a new vehicle record. Payload must include model, year, plate, latitude, longitude, and imageUrl. Car (body) 201 · Car
PATCH /car/{id} Partial update of an existing vehicle. Only provided fields are mutated; absent fields retain their current values. Partial<Car> 200 · Car
DELETE /car/{id} Permanently removes the vehicle record identified by {id}. Idempotent — repeated calls return 404 after the first successful deletion. — 204 · No body

Security Model

Security & Data Protection

Security decisions were made deliberately across the authentication layer, image storage access model, and network communication stack.

🔑
Credential-Free Authentication
Firebase Phone Auth · OTP/TOTP
No passwords are stored, transmitted, or managed by the application. The OTP is single-use, carrier-verified, and expires in 60 seconds. This eliminates the most common attack vectors: credential stuffing, brute-force, and phishing of static passwords.
🗝️
JWT Token Hygiene
RS256 · 1h TTL · Refresh Token
ID tokens are short-lived JWTs (1h) signed asymmetrically. The SDK manages rotation transparently. The app never persists raw tokens to disk — only FirebaseAuth's internal encrypted storage is used, backed by Android's EncryptedSharedPreferences or Keystore on API 23+.
🪣
Firebase Storage Access Rules
Storage Security Rules · Per-path ACLs
Storage Security Rules are configured to restrict write operations to authenticated users only (request.auth != null). Read access to car images is public, which allows efficient URL-based rendering via Picasso without re-authentication on each image fetch.
📡
Network Transport Security
TLS · OkHttp · Network Security Config
All Firebase and Google Maps traffic is enforced over TLS 1.2+. The local API (development/LAN) runs over HTTP; in production this would be placed behind an HTTPS reverse proxy (nginx/Caddy). OkHttp Logging Interceptor is strictly enabled only in debug build variants to prevent credential leakage in production logs.
🔒
API Key Isolation
AndroidManifest · google-services.json
The Google Maps API key is declared in AndroidManifest.xml and restricted by package name and SHA-1 fingerprint in the Google Cloud Console. The google-services.json is excluded from version control via .gitignore.
🚫
State-Based Access Control
AuthStateListener · Activity stack
A global FirebaseAuth.AuthStateListener ensures that any unauthenticated state (logout, token revocation) immediately redirects to the login screen with the back stack cleared, preventing unauthorized re-entry via Android's back navigation.
Production note: For a production deployment, the backend API should implement Firebase Admin SDK token verification as middleware — validating the ID token on every mutating request to enforce server-side access control, not just client-side state.

Implementation Details

Image Upload Pipeline

Uploading a vehicle image involves three sequential, asynchronous stages. The final persisted record contains only the durable download URL — not the raw binary data.

Kotlin · CarFormActivity.kt — Firebase Storage upload 
// Stage 1: Build a StorageReference under a unique path per vehicle
val storageRef = FirebaseStorage.getInstance()
    .reference
    .child("cars/${UUID.randomUUID()}.jpg")

// Stage 2: Upload the selected bitmap as a byte stream
val uploadTask = storageRef.putBytes(imageBytes)

uploadTask
    .continueWithTask { task ->
        if (!task.isSuccessful) throw task.exception!!
        storageRef.downloadUrl        // Stage 3: resolve the permanent CDN URL
    }
    .addOnCompleteListener { task ->
        if (task.isSuccessful) {
            val imageUrl = task.result.toString()
            viewModel.saveCarWithImage(carData, imageUrl)  // POST to REST API
        } else {
            viewModel.setUploadError(task.exception?.message)
        }
    }
