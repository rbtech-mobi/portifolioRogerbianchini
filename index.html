What it is

A reusable SaaS where each tenant is an independent food business. Customers browse a responsive public catalog, build a cart, and place orders; owners manage their menu, fulfil orders through a state machine, record payments, and customize their brand — all from an installable Progressive Web App. The single most important invariant is strict tenant isolation: no business can ever read or mutate another's data.

Decisions worth defending
Money as integer cents
Prices, totals, and refunds are stored and computed as integers end-to-end. Binary floating point never touches money — the browser only divides by 100 for display.
Append-only payment ledger
A refund is a counter-entry, not an edit. Current financial status is derived from the event log, which stays the audit source of truth.
Authentication ≠ authorization
Firebase proves identity; a database lookup grants permission. A valid Google account with no admin_users row gets a clean 403.
Idempotency for orders
An Idempotency-Key plus a unique index means a double-submit returns the original order instead of charging stock twice.
Untrusted uploads re-encoded
Uploads are decoded and re-written as WebP. A file that isn't a real image is rejected; original bytes are never served.
Migration before code
Schema changes ship as forward-only migrations that must run before dependent code. Features needing new columns are staged, not rushed.
Security posture

Server-side tenant isolation · parameterized SQL throughout · Firebase token verification · secrets only in environment variables · Helmet CSP tuned for the auth flow · rate limiting · validated image re-encoding · XSS-safe rendering of owner content · cross-tenant lookups indistinguishable from "not found" · dependency advisories resolved to zero.

+ How it works & the sales logic behind it
What it is

A frontend built to sit inside a sales workflow, not next to it. It ingests context about a prospect directly from monday.com — via webhook or native automation — and if no data has arrived yet, it falls back to a manual entry mode so an SDR can still use it cold.

What it vectorizes

Structured: prospect name, role, company, website, and the specific offer being pitched.
Unstructured (NLP): free-text research notes — a recent LinkedIn post, a comment, company news, any buying signal a rep found manually.

What it does with it

On "Analyze prospect," the tool sends the combined structured + unstructured payload to a connected LLM, which cross-references the prospect's real pain points against the seller's value proposition — and returns an approach angle calibrated to that specific person, not a template.

Worked example

An SDR selling compliance software pastes in a LinkedIn post where the prospect — a VP of Product — is complaining about compliance friction across a remote team. The tool doesn't generate a generic pitch; it generates an opener that references that exact friction, because the offer was mapped to a real, current pain point instead of a job title.

Why this is a sales artifact, not just a tool

This is the part of the job I actually understand from the inside: I know what makes an SDR skip research, and I know what a rep needs in front of them to send a message that gets a reply. I built the tool I wish I'd had.

+ Hardware, firmware architecture & MQTT contract
What it is

An interactive desktop companion running bare-metal C++ on a WT32-SC01 Plus (ESP32-S3, 3.5" touch display). It subscribes to an n8n orchestration server over MQTT, surfaces the message on a touch terminal, and lets the operator tap a response — SIM / NAO / REVERTA / PARE — that publishes straight back to the broker. Idle, it becomes an animated Back to the Future-themed time-circuit clock.

MQTT contract

Subscribes: home/llm/inbound (QoS 1) — messages rendered in the terminal.
Publishes: home/llm/response — operator's selected action · home/llm/status (retained) — device health heartbeat every 45s, so the broker always knows if the device is alive.

Why it matters beyond the novelty

This proves I can go past the API layer — GPIO, I2C, a parallel display bus, TLS certificate handling, and a real reconnection strategy for a network that will drop. It's the same reliability thinking as the SaaS ledger, just applied to a device instead of a database.
